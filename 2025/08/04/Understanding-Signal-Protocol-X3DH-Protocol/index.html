前言

在數位時代，科技公司經常蒐集用戶的個人資訊，而 Signal 憑藉其強大的端對端加密（End-to-End Encryption, E2EE）技術而受到矚目，將用戶隱私放在首位。該應用程式獲得了多位關注隱私權的知名人士支持，例如愛德華·斯諾登（Edward Snowden）、馬克·祖克柏（Mark Zuckerberg）及伊隆·馬斯克（Elon Musk），成為重視通訊隱私人士的首選工具。

簡介

在典型的即時通訊應用中，用戶透過「可信任」的伺服器進行通訊，伺服器負責接收並轉發訊息。這樣的架構要求用戶對伺服器高度信任，預期其不會濫用或外洩資料。然而，若伺服器遭到入侵，攻擊者可能竊取所有未加密的訊息內容。

Signal 協議（Signal Protocol）是一種開源的端對端加密協議，旨在確保訊息的明文僅存在於通訊雙方的客戶端設備上。即使伺服器被攻擊，攻擊者也無法解密訊息。此外，Signal 協議具備前向安全性（Forward Secrecy）與後向安全性（Backward Secrecy），確保歷史通訊在密鑰洩露後依然保密。

用戶註冊時的密鑰生成

迪菲-赫爾曼密鑰交換算法（Diffie-Hellman Key Exchange）

要理解 Signal 協議的端對端加密機制，需先瞭解迪菲-赫爾曼密鑰交換算法（Diffie-Hellman Key Exchange, DH）。此算法允許雙方在不直接傳輸秘密的情況下，透過公開通道建立共享秘密（Shared Secret）。其安全性基於離散對數問題（Discrete Logarithm Problem）的計算困難性。

假設有兩位用戶 Alice 和 Bob，以及一個公開伺服器，伺服器儲存以下兩個參數：

  • g：一個小的質數，作為生成元（Generator）。
  • n：一個非常大的質數（通常為 2000 至 4000 位元），確保計算複雜度。

密鑰交換流程如下：

  1. 生成私鑰（Private Key）：Alice 和 Bob 各自生成私鑰 a 與 b，其中 1 ≤ x ≤ n。
  2. 生成公鑰（Public Key）：
    • Alice 計算公鑰為 (g ^ a) % n
    • Bob 計算公鑰為 (g ^ b) % n
  3. 交換並計算共享秘密：
    • Alice 使用 Bob 的公鑰計算 ((g ^ b) % n) ^ a % n
    • Bob 使用 Alice 的公鑰計算 ((g ^ a) % n) ^ b % n

由於模冪運算的交換性質，兩者所得結果相同，成為雙方共享的秘密。

以下為 TypeScript 實作範例：

// 假設 g = 11, n = 100（實際應用中 n 應為大質數）
const g = 11;
const n = 100;

const a = randomIntFromInterval(1, n); // Alice 的私鑰
const b = randomIntFromInterval(1, n); // Bob 的私鑰

const a_key = modExp(g, a, n); // Alice 的公鑰
const b_key = modExp(g, b, n); // Bob 的公鑰

const a_shared_secret = modExp(b_key, a, n); // Alice 的共享秘密
const b_shared_secret = modExp(a_key, b, n); // Bob 的共享秘密

console.log(`共享秘密是否相同：${a_shared_secret === b_shared_secret}`); // true

function randomIntFromInterval(min: number, max: number) {
  return Math.floor(Math.random() * (max - min + 1) + min);
}

function modExp(base: number, exponent: number, modulus: number) {
  if (modulus === 1) return 0;
  let result = 1;
  base = base % modulus;
  while (exponent > 0) {
    if (exponent % 2 === 1) {
      result = (result * base) % modulus;
    }
    exponent = Math.floor(exponent / 2);
    base = (base * base) % modulus;
  }
  return result;
}

X3DH 密鑰協議（Extended Triple Diffie-Hellman）

擴展三重迪菲-赫爾曼（Extended Triple Diffie-Hellman, X3DH）是專為非同步通訊場景設計的密鑰交換協議。由於用戶可能在不同時間上線，傳統 DH 協議無法直接應用，X3DH 透過預儲密鑰（Prekeys）實現靈活且安全的初始密鑰協議。

發佈預密鑰（Prekeys）

當 Bob 註冊 Signal 時，會上傳以下資訊至伺服器：

  • 身份密鑰（Identity Key, IKB）：長期公鑰，用於身份驗證。
  • 簽名預密鑰（Signed Prekey, SPKB）：有效期中長的公鑰，需定期輪替。
  • 預密鑰簽名（Sig(IKB, SPKB)）：使用 IKB 對 SPKB 的簽名。
  • 一次性預密鑰（One-Time Prekeys, OPKB）：供一次性使用的公鑰組。

傳送訊息

當 Alice 欲向 Bob 發送訊息時，會從伺服器取得 Bob 的預密鑰包（Prekey Bundle），其中包含：

  • Bob 的身份密鑰（IKB）
  • Bob 的簽名預密鑰（SPKB）
  • 預密鑰簽名（Sig）
  • 一組一次性預密鑰（OPKB）

驗證預密鑰包

Alice 使用 Bob 的身份密鑰驗證 SPKB 的簽名，確認密鑰未遭竄改且來自 Bob。

生成臨時密鑰（Ephemeral Key）

Alice 產生一組臨時密鑰（EKA），僅用於此次通訊。

計算共享密鑰

Alice 執行下列四次 DH 計算：

DH1 = DH(IKA, SPKB) // Alice 的身份私鑰與 Bob 的簽名預密鑰進行 DH 運算。
DH2 = DH(EKA, IKB) // Alice 的臨時私鑰與 Bob 的身份公鑰進行 DH 運算。
DH3 = DH(EKA, SPKB) // Alice 的臨時私鑰與 Bob 的簽名預密鑰進行 DH 運算。
DH4 = DH(EKA, OPKB) // Alice 的臨時私鑰與 Bob 的一次性預密鑰進行 DH 運算。
KAB = KDF(DH1 || DH2 || DH3 || DH4) // 這些運算結果串聯並透過密鑰衍生函數（Key Derivation Function, KDF）生成共享密鑰KAB
運算 目的
DH1、DH2 確保雙方身份驗證
DH3、DH4 提供前向安全性（DH3 和 DH4 使用短期密鑰）
DH4 支援非同步通訊（DH4 使用一次性預密鑰）
KDF 增強密鑰多樣性和安全性（多重 DH 結果串聯）

發送初始訊息

Alice 使用共享密鑰 KAB 加密訊息，並一併傳送以下資訊：

  • Alice 的身份密鑰（IKA）
  • Alice 的臨時密鑰（EKA）
  • Bob 的一次性預密鑰索引
  • 加密後的初始密文

訊息解密

Bob 接收訊息後，執行相同的 DH 計算程序產生 KAB，解密訊息並驗證其正確性。使用後的一次性預密鑰會被刪除，進一步確保訊息無法在未來被重複解密。

結論

Signal 協議透過 X3DH 密鑰協議，在公開伺服器的前提下安全建立共享密鑰，實現真正的端對端加密。其前向與後向安全特性使歷史訊息在密鑰洩露後仍能保密。此協議目前已廣泛應用於包括 Signal、WhatsApp 在內的通訊平台。
